Cross-Project Agent Coordination

How an MSP runs 24 codebases on one orchestrator — and why the alternative is a 41–86% failure rate.

The problem nobody is fully solving

In late 2025, a Berkeley/CMU research group published MAST — the Multi-Agent Systems Taxonomy. It validated against 1,600+ execution traces across the major orchestration frameworks: AutoGen, CrewAI, LangGraph, OpenAI Swarm. The finding: production multi-agent LLM systems fail at rates between 41% and 86.7%, depending on the framework and task class.

The failure modes were not bugs you could grep for. They clustered into three roots:

  1. Specification ambiguity. Agent A and Agent B both think they own the same step. Or neither does.
  2. Coordination breakdown. Agent A waits on a tool result that Agent B already cached, never receives the message, retries, double-spends.
  3. Verification gaps. Agent A hallucinates a fact. Agent B accepts it as input and reasons forward. By the time the human reviews the output, "false consensus" is locked in — and there’s no stack trace, because every individual hop succeeded.

And the transport layer is on fire

Anthropic’s Model Context Protocol (MCP) became the de facto wiring for multi-agent systems in 2025. By Q1 2026 it had been donated to the Linux Foundation under a new Agentic AI Foundation, with Block, OpenAI, Google, Microsoft, AWS, and Cloudflare on the steering committee. Adoption is real: 78% of enterprise AI teams have at least one MCP agent in production. There are over 17,000 indexed MCP servers and more than 110 million SDK downloads per month.

It is also actively under attack. Since GA, MCP has accumulated more than 30 public CVEs. The exploit classes that matter for an MSP:

  • Tool poisoning. A malicious Jira ticket or Slack message containing carefully-crafted text gets retrieved by an agent, and the resulting tool call is hijacked.
  • JWT leakage. Stdio-mode MCP servers default to no authentication. Tokens get logged. Tokens get checked into git. Tokens get reused across tenants.
  • STDIO RCEs. Several popular MCP servers shipped with unsafe deserialization defaults. Ten remote-code-execution-class issues are public as of May 2026.
  • Idempotency, not optional. The protocol does not specify retry semantics. If a deploy agent runs twice because the orchestrator timed out, you ship the same release twice.

This is the AI analogue of the early-2000s SOAP / WS-Security moment. The protocol works, the tooling is everywhere, and the security posture is uneven by default.

Why generic frameworks won’t save you

LangGraph (now 80,000+ GitHub stars), CrewAI (44,600+), Microsoft Agent Framework (the merged Semantic Kernel + AutoGen), Google’s ADK with its A2A protocol — all are excellent for building single-tenant agentic applications. None solve the problems an MSP actually has, because none of them assume a multi-tenant deployment with strict cross-tenant isolation, audit-grade evidence collection, and lock arbitration across independent codebases.

An MSP runs many environments simultaneously. We currently coordinate across twenty-four products — everything from a Slack bot to a SOC platform to a building-controls analytics service to multiple client-deployed integrations. Two of those products will, on any given day, want to scan, deploy to, or migrate the same client repo. Without arbitration, the deploy and the scan race; the scan reads partial state; the deploy succeeds; the audit trail is incoherent.

The frameworks above do not arbitrate that. They are not built to.

What we built

We built a coordination layer specifically for the MSP shape: AiT Coord. It is a JWT-signed multi-agent orchestrator exposed over MCP, deployed on Cloud Run, and operating against a small number of well-defined primitives:

  • Project locks. Before any agent acts on a project (scan, deploy, migrate, fix), it acquires a project-scoped lock with a TTL and an intent. Conflicting intents return a structured conflict response with a reason. No agent ever assumes ownership of state they didn’t lock.
  • Per-agent identity. Every agent is issued a short-lived JWT signed with an RS256 key in Google Secret Manager. Tenant ID is a claim. The orchestrator verifies on every call. Non-Human Identity sprawl — the perimeter problem 78% of organizations admit they don’t govern — collapses to one verifiable signing key per environment.
  • Idempotent tool calls. Every tool exposed through coord carries an idempotency key. Retries are safe. Crashes during a multi-step operation can resume.
  • Audit trail by default. Every lock acquisition, tool call, and subagent spawn writes a row to an agent_runs table. The Trust Portal exposes this to clients on request. Auditors get an answer in minutes, not weeks.
  • Subagent spawn for high-severity findings. When our continuous audit system finds a critical vulnerability in a client repo, it asks coord to spawn a fixer subagent — which inherits the caller’s tenant scope, opens a PR, and releases the lock. The human still reviews. The mechanical work is done.
In production today

AiT Coord runs at ait-agent-coord-jcxklz7ilq-uc.a.run.app. It coordinates AiTAgent (Slack), AiT Teams Bot, AiT Audit, the MCP bridges on every developer machine, and an expanding set of fixer subagents. It is the spine that lets one team operate twenty-four codebases without race conditions.

What this looks like in practice

A worked example. Last week, our continuous-audit system began a monthly scan against twenty-four MRoot2025 repos. Halfway through, a deploy was triggered against one of them — ait-crm — by a separate session.

Without coordination, the scan would have run against a half-deployed working tree. Findings would have referenced files that no longer existed. The audit would have either failed silently or, worse, reported a false-positive set that contaminated the trend chart.

What actually happened: the audit acquired a 10-minute scan lock against ait-crm. The deploy attempted to acquire a deploy lock against the same project, was returned {state: "conflict", reason: "scan in progress"}, and queued itself. The audit completed in 4 minutes, released the lock, and the deploy ran cleanly thereafter. Both events are rows in agent_runs; the Trust Portal can show a customer their full history of who-did-what across their environment with two clicks.

What you should ask your current MSP

  1. How do your agents prove their identity to each other, and what cryptographic primitive does that depend on?
  2. Show me the audit log of every agent action that touched my environment in the last 30 days.
  3. What happens if two of your agents try to act on my system at the same time?
  4. Which of your agents inherit which tenant scope, and how would you prove that to an auditor?

Most MSPs cannot answer these. Most agentic-AI vendors cannot either. This is the gap that opens after the agent demo wraps and the production deployment begins.
